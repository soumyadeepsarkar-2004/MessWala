    Community Guidelines¶

    Welcome to the MessWala Community! We're excited to have you here. These guidelines help us maintain a friendly, respectful, and productive community.

    Our Community Values¶

    • Respect - Treat all members with kindness and respect
    • Openness - Welcome diverse perspectives and ideas
    • Collaboration - Work together to improve MessWala
    • Excellence - Strive for high-quality contributions
    • Transparency - Communicate openly and honestly

    Getting Started¶

    Ways to Contribute¶

    1. Report Bugs - Help us identify and fix issues
    2. Suggest Features - Share your ideas for improvements
    3. Write Code - Contribute code improvements
    4. Improve Docs - Help documentation stay current
    5. Answer Questions - Help other community members
    6. Share Feedback - Let us know what you think

    First Time Contributing?¶

    1. Check out our CONTRIBUTING.md
    2. Look for "good first issue" labels
    3. Join our Discussions
    4. Ask questions in issues or discussions - we're here to help!

    Communication Channels¶

    GitHub Issues & Discussions¶

    • Purpose: Bug reports, feature requests, technical questions
    • When to use: Technical topics, code-related questions
    • Response time: 24-48 hours

    GitHub Discussions¶

    • Purpose: General questions, ideas, announcements
    • When to use: Brainstorming, best practices, non-urgent topics
    • Response time: 1-3 days

    Email¶

    • Purpose: Security issues, conduct concerns
    • Email: soumyadeep.sarkar@example.com

    Community Code of Conduct¶

    Please refer to our full Code of Conduct for detailed guidelines.

    Key Principles¶

    • No harassment, discrimination, or hate speech
    • Respect others' ideas and opinions
    • Assume good intent
    • Give constructive feedback
    • Ask before giving unsolicited advice

    Best Practices¶

    Asking Questions¶

    ✅ DO: - Search existing issues/discussions first - Provide context and relevant details - Use clear, descriptive titles - Share error messages and logs

    ❌ DON'T: - Ask the same question multiple times - Post the same message in multiple channels - Use all caps or excessive punctuation - Demand urgent responses

    Giving Feedback¶

    ✅ DO: - Be specific and constructive - Suggest improvements - Acknowledge good work - Ask clarifying questions

    ❌ DON'T: - Give vague criticism - Attack the person instead of the idea - Use sarcasm or mockery - Assume malicious intent

    Reporting Issues¶

    ✅ DO: - Use the bug report template - Include steps to reproduce - Share environment details - Provide error messages

    ❌ DON'T: - Report sensitive information publicly - Use offensive language - Cross-post identical issues - Expect immediate fixes

    We value all contributions! Contributors will be:

    • Added to CONTRIBUTORS.md
    • Mentioned in release notes
    • Given contributor badge on GitHub
    • Invited to monthly community calls

    • Duration: 1 hour
    • Topics: Roadmap updates, Q&A, feature discussions
    • Join: [Link on Discussions]

    Weekly Office Hours¶

    • When: Wednesdays, 6:00 PM IST
    • Duration: 30 minutes
    • Purpose: Help with setup, questions, debugging
    • Open to: All community members

    Seasonal Hackathons¶

    • When: Quarterly
    • Focus: New features, bug fixes, documentation
    • Prizes: Featured contributor status, swag

    Resolving Conflicts¶

    If You Disagree¶

    1. Listen - Try to understand the other perspective
    2. Discuss - Share your perspective respectfully
    3. Escalate - Ask a maintainer to help mediate
    4. Accept - Respect the final decision

    If There's a Code of Conduct Violation¶

    1. Document - Save screenshots/messages
    2. Report - Email conduct concerns (see above)
    3. Follow-up - Allow time for investigation
    4. Support - We're here to help if you need it

    How We Make Decisions¶

    Small Changes (typos, docs, small bug fixes)¶

    • Reviewed and merged by any maintainer
    • Can happen quickly

    Medium Changes (new features, refactoring)¶

    • Discussion in issues/PRs
    • Reviewed by 2+ maintainers
    • Requires maintainer approval

    Large Changes (architecture, major features)¶

    • Community discussion period (1-2 weeks)
    • RFC (Request for Comments) discussion
    • Decided by project lead

    Code Review Process¶

    All contributions go through code review:

    1. Automated Checks - Linting, testing, builds pass
    2. Manual Review - Maintainer reviews code quality
    3. Community Feedback - Community members can comment
    4. Revision - Author addresses feedback
    5. Approval - Maintainer approves
    6. Merge - Code merged to main branch
